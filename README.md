## рЯІ† Architecture Decisions

> Why this stack? Every choice is intentional.

**AgentHub** вАФ pgvector inside PostgreSQL instead of a dedicated vector DB вАФ single data layer, transactional consistency, simpler backups. Semantic cache in Redis with LSH bucketing: near-identical questions return cached answers without burning tokens. MCP server for interoperability вАФ external agents can use AgentHub's knowledge base as a tool.

**EventPipe** вАФ Kafka over RabbitMQ because the pipeline processes high-throughput event streams where ordering and replay matter. Separate Ingest/Transform/Query services with different scaling profiles. Dead Letter Queue as a separate topic вАФ failed events don't block the main pipeline.

**TaskFlow** вАФ Django for ORM-heavy domain logic + FastAPI for the async notification consumer. RabbitMQ topic exchange decouples write-side transactions from delivery.

**URLShort** вАФ Redis sliding window (ZSET) for rate limiting вАФ exact per-second precision. Atomic counters + periodic flush reduce PG write load by ~100x. Nginx microcaching for 301 redirects вАФ hot URLs served without hitting FastAPI.

**WebHook Manager** вАФ Celery for fire-and-forget delivery. Circuit breaker prevents hammering dead endpoints. HMAC + idempotency at infrastructure layer, not business logic.

**RealTimeChat** вАФ Redis Pub/Sub over in-memory broadcast вАФ scaling to N replicas is transparent. WS auth via query param because browsers don't support custom headers on handshake.
